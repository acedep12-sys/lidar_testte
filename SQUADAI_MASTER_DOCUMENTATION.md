# Comprehensive Plan & Goals to Fix the SquadAI + TPWCS System

This document outlines the executive summary, architectural diagnosis, phased execution plan, and concrete engineering goals to permanently resolve the integration between **SquadAI v3** (Custom C++ AI/Combat/Mission Engine) and **TPWCS V2** (Third Person Weapon & Combat System Pack).

---

## 1. The Core Integration Goal

To build a unified open-world shooter system where:
1. **SquadAI C++ owns 100% of the Brain**: Mission movement, cover scanning (`CoverSystemSubsystem`), perception fusion (`PerceptionMemory`), squad leader pacing, tactical StateTree decisions, and damage registration.
2. **TPWCS V2 owns 100% of the Presentation**: AAA weapon animations, firing montages, muzzle flashes, bullet tracers, hit reactions, and left-hand IK grip stabilization.
3. **Zero Manual Blueprint Hacking**: AI characters and controllers generated by the automated asset factory work immediately upon spawning without manual Blueprint wiring.

---

## 2. Why Previous Attempts Failed (The Diagnosis)

Earlier integration attempts suffered from silent Blueprint failures, animation corruption, and missing firing visuals due to four root issues:

* **1. Missing Aiming State Preconditions (`CII_StartAiming`)**:  
  TPWCS V2 weapon firing Blueprints (`CII_StartFiring`) evaluate internal branch checks on `aimingState`. If an AI character was not transitioned into `Aiming` state beforehand, the Blueprint silently blocked fire animations and projectile spawns.
* **2. Bypassing Blueprint Event Gates via Reflection**:  
  Attempting to trigger visual fire by simply setting properties (`bIsShooting = true`) bypassed TPWCS internal function gates (`Plr_SetFiringInputKey` $\rightarrow$ `Char_OpenFiringGate`).
* **3. Blueprint Hierarchy Wipeout (Reparenting Trap)**:  
  Reparenting `BP_AICharacter` directly to C++ `AEnemySoldier` destroyed its connection to the pack's `BP_Character` parent graph, losing essential variable defaults and component hooks.
* **4. Blind AI Controllers**:  
  TPWCS weapon components query the AI Controller for targeting data via `BPC_AIDetection_Master`. Standard SquadAI controllers (`ASoldierAIController`) lacked this component by default.

---

## 3. The 3-Phased Fix Plan

### Phase 1 — Restore Compile & Stable Gameplay Authority
* **Goal**: Guarantee zero C++ compilation errors, balanced braces, and rock-solid core gameplay intelligence.
* **Deliverables**:
  * Clean, balanced C++ implementation in `SoldierAIController.cpp` and `SquadAIAdapterComponent.cpp`.
  * AI reliably moves to quest zones, detects enemies, claims cover, and delivers gameplay damage without crashing or corrupting animations.

### Phase 2 — Controller & Hierarchy Alignment
* **Goal**: Ensure AI controllers and generated characters automatically carry required TPWCS components and parent chains.
* **Deliverables**:
  * **Automated Controller Detection**: `ASoldierAIController::OnPossess()` dynamically verifies and attaches `BPC_AIDetection_Master` at runtime if missing.
  * **Sandbox Parent Hierarchy**: Asset generation duplicates `BP_Character` as a base class (`BP_Enemy_PackClean_Auto_PackBase`) before parenting `BP_AICharacter` templates, keeping 100% of pack Blueprint functions intact while satisfying C++ type checks (`IsA(AEnemySoldier)`).

### Phase 3 — TPWCS Visual Authority & Exact Event Flow
* **Goal**: Transfer visual firing, weapon IK, and muzzle presentation entirely to TPWCS V2.
* **Deliverables**:
  * **Option C Robustness Pipeline**: When C++ commands a burst, `SquadAIAdapterComponent::StartVisualFire()` enforces the exact verified Blueprint execution order:
    1. `Invoke("CII_StartAiming")` & `SetAimingStateProperty(Target, true)`
    2. `SyncAimData(Target, AimLocation, AimActor)`
    3. `Invoke("Char_OpenFiringGate")`
    4. `InvokePackKeyFunction(Target, "Plr_SetFiringInputKey", EKeys::LeftMouseButton)`
    5. `Invoke("CII_StartFiring")`
  * **Explicit Observability**: Null pack weapons immediately log clear, actionable warnings instead of silently failing.

## 1. Executive Summary: Why Screenshot Mining Replaces Guessing

Earlier integration efforts suffered from trial-and-error firing triggers that caused animation corruption or missing visuals (muzzle flashes, bullet tracers, montages). Systematic mining of the actual pack and project screenshots reveals two foundational facts about the TPWCS pack:

1. **Input & Gate Flow:** Firing visuals and weapon mechanics in TPWCS are gated behind explicit blueprint function calls that pass struct parameters (specifically `FKey`), rather than simple property assignments (`bIsShooting = true`).
2. **Controller Separation:** SquadAI controllers (`BP_LeaderAIController`, `BP_AllyAIController`) do not natively include TPWCS detection components (`BPC_AIDetection_Master`), requiring deliberate runtime injection or template alignment.

## 4. Exact Pack Firing Input Flow

### Observations from Firing Graph Screenshots (`2026-07-01 021810.png`, `2026-07-01 035530.png`)
The Event Graph for player and character weapon firing shows the exact sequential execution order required to initiate visual firing (muzzle flashes, sound effects, bullet tracers, and firing montages):

#### Start Firing Sequence (Input Pressed)
1. **`Plr_SetFiringInputKey(Key)`** — Sets the internal input key struct (e.g., `EKeys::LeftMouseButton`) and alerts the weapon subsystem that a fire key is active.
2. **`Char_OpenFiringGate()`** — Opens the internal blueprint gate permitting continuous/burst firing loops.
3. **`CII_StartFiring()`** — Executes the interface/component firing routine, spawning projectiles and triggering weapon montages.

#### Stop Firing Sequence (Input Released)
1. **`Plr_SetFiringInputKey(None)`** — Resets the input key struct to an empty `FKey()`.
2. **`CII_StopFiring()`** — Terminate continuous firing loops.
3. **`Char_CloseFiringGate()`** — Closes the internal blueprint gate.

### Why Direct Property Assignment Failed
Previous bridge code attempted:
```cpp
SetBool(Target, { FName("bIsShooting?") }, true);
SetKey(Target, { FName("firingInputKey") }, EKeys::LeftMouseButton);
Invoke(Target, "CII_StartFiring");
```
Because `Plr_SetFiringInputKey` was never executed via `ProcessEvent`, the Blueprint internal event handlers that depend on the function execution did not fire, resulting in silent/missing visual fire.

## 5. C++ Adapter Bridge Patch (`SquadAIAdapterComponent.cpp`)

To adhere strictly to the screenshot-verified execution flow without crashing or guessing, `SquadAIAdapterComponent.cpp` implements a dedicated `FKey` function invocation helper:

### 5.1 Dedicated Reflection Helper
```cpp
static bool InvokePackKeyFunction(UObject* Target, FName FunctionName, const FKey& KeyValue)
{
	if (!Target || FunctionName.IsNone())
	{
		return false;
	}

	UFunction* Function = Target->FindFunction(FunctionName);
	if (!Function)
	{
		return false;
	}

	TArray<uint8> Params;
	Params.SetNumZeroed(Function->ParmsSize);
	void* ParamsBuffer = Params.Num() > 0 ? Params.GetData() : nullptr;

	for (TFieldIterator<FProperty> It(Function); It && It->HasAnyPropertyFlags(CPF_Parm); ++It)
	{
		FProperty* Property = *It;
		if (!Property || Property->HasAnyPropertyFlags(CPF_ReturnParm | CPF_OutParm))
		{
			continue;
		}
		if (const FStructProperty* StructProp = CastField<FStructProperty>(Property))
		{
			if (StructProp->Struct == FKey::StaticStruct())
			{
				if (ParamsBuffer)
				{
					*StructProp->ContainerPtrToValuePtr<FKey>(ParamsBuffer) = KeyValue;
				}
				break;
			}
		}
	}

	Target->ProcessEvent(Function, ParamsBuffer);
	return true;
}
```

### 5.2 Synchronized Firing Execution
In `StartVisualFire()`:
```cpp
SyncAimData(Target, AimLocation, AimActor);
SetBool(Target, { FName(TEXT("bIsShooting?")), FName(TEXT("bIsShooting")), FName(TEXT("IsShooting")) }, true);
SetKey(Target, { FName(TEXT("firingInputKey")), FName(TEXT("FiringInputKey")) }, EKeys::LeftMouseButton);

Invoke(Target, TEXT("Char_OpenFiringGate"));
InvokePackKeyFunction(Target, TEXT("Plr_SetFiringInputKey"), EKeys::LeftMouseButton);
Invoke(Target, TEXT("CII_StartFiring"));
```

In `StopVisualFire()`:
```cpp
SetBool(Target, { FName(TEXT("bIsShooting?")), FName(TEXT("bIsShooting")), FName(TEXT("IsShooting")) }, false);
SetKey(Target, { FName(TEXT("firingInputKey")), FName(TEXT("FiringInputKey")) }, FKey());

InvokePackKeyFunction(Target, TEXT("Plr_SetFiringInputKey"), FKey());
Invoke(Target, TEXT("CII_StopFiring"));
Invoke(Target, TEXT("Char_CloseFiringGate"));
```

# Animation Blueprint Automation Guide

Date: 2026-06-19

## Purpose

The goal is to minimize Blueprint math and make soldier animation setup mostly data-driven from C++ helper functions.

Instead of manually calculating speed, direction, cover movement, aim offsets, and weapon IK in the AnimBP, use a single helper:

```cpp
ASoldierCharacter::GetSoldierAnimRuntimeData(...)
```

---

## Files Changed

```text
Source/SquadAI/Public/Characters/SoldierCharacter.h
Source/SquadAI/Private/Characters/SoldierCharacter.cpp
```

---

## New Runtime Data Struct

Added:

```cpp
FSoldierAnimRuntimeData
```

It contains:

```cpp
float Speed;
float Direction;
bool bIsAlive;
bool bIsCrouched;
bool bCombatReady;
bool bWantsToFire;

bool bIsInCover;
bool bIsCrouchingInCover;
bool bIsPeekingFromCover;
bool bCoverIsLow;
bool bCoverSideIsLeft;
bool bCoverPeekIsOver;
bool bIsMovingInCover;
bool bIsAdjustingCover;
float CoverMoveDirection;

float AimYaw;
float AimPitch;

FWeaponIKRuntimeData WeaponIK;
```

---

## New AnimBP Helper Function

Added:

```cpp
FSoldierAnimRuntimeData GetSoldierAnimRuntimeData(
    float CoverMoveSpeedThreshold = 20.f,
    float CoverAdjustDirectionThreshold = 0.35f,
    float CombatThreatConfidence = 0.25f
) const;
```

This function computes:

- locomotion speed,
- locomotion direction,
- alive/crouch/combat state,
- cover booleans,
- cover side/peek type,
- moving-in-cover state,
- cover move direction,
- cover adjustment state,
- aim yaw/pitch from `AimComp->GetSpineAimOffset()`,
- weapon IK runtime data.

## Recommended AnimBP Event Graph Setup

In `ABP_CC_Soldier_Basic`:

1. Keep `Try Get Pawn Owner`.
2. Cast to `SoldierCharacter`.
3. From `As SoldierCharacter`, call:

```text
Get Soldier Anim Runtime Data
```

4. Break the returned struct.
5. Set your AnimBP variables from it.

Recommended AnimBP variables:

```text

Direction : float
bIsAlive : bool
bIsCrouched : bool
bCombatReady : bool
bWantsToFire : bool

bIsCrouchingInCover : bool
bIsPeekingFromCover : bool
bCoverIsLow : bool
bCoverSideIsLeft : bool
bCoverPeekIsOver : bool
bIsMovingInCover : bool
bIsAdjustingCover : bool
CoverMoveDirection : float

AimYaw : float
AimPitch : float

bEnableLeftHandIK : bool
LeftHandIKAlpha : float
LeftHandIKLocation_CS : vector
LeftHandIKRotation_CS : rotator
LeftElbowJointTarget_BS : vector
bUseLeftHandRotation : bool
LeftHandRotationAlpha : float
```

From `WeaponIK` inside the struct:

```text
WeaponIK.bEnabled                 → bEnableLeftHandIK
WeaponIK.Alpha                    → LeftHandIKAlpha
WeaponIK.LeftHandEffectorLocationCS → LeftHandIKLocation_CS
WeaponIK.LeftHandEffectorRotationCS → LeftHandIKRotation_CS
WeaponIK.LeftElbowJointTargetBS     → LeftElbowJointTarget_BS
WeaponIK.bUseLeftHandRotation       → bUseLeftHandRotation
WeaponIK.LeftHandRotationAlpha      → LeftHandRotationAlpha
```

This replaces previous Blueprint calculations such as:

- Get Velocity → Vector Length XY,
- Calculate Direction,
- manual CoverMoveDirection dot product,
- manual AimComp GetSpineAimOffset,
- manual weapon socket transform conversion.

Before this helper, the AnimBP needed many nodes:

```text

Direction math
Cover booleans
Cover enum comparisons
Cover move direction dot product
Aim offset extraction
```

Now all of that is centralized in C++ and exposed as one Blueprint node.

```text
→ Convert Component To Local Space
```

---

## Tuning Order

1. Tune `WeaponAttachLocationOffset` / `WeaponAttachRotationOffset` until rifle sits correctly in right hand and stock does not clip badly.
2. Move weapon `LeftHandIK` socket to the **wrist target** position, not palm/finger surface.
3. Tune `LeftElbowJointTarget_BoneSpace` until elbow bends naturally.
4. Tune `LeftHandIKEffectorLocationOffset_CS` for small left hand placement corrections.
5. Only then enable `bUseLeftHandIKRotation` and tune `LeftHandIKRotationOffset` / weapon socket rotation.
6. Add a finger/grip pose later. IK does not curl fingers.

---

## Important Concept

`hand_l` is usually the wrist joint. Therefore the weapon `LeftHandIK` socket should be placed where the **wrist** should be, not where the palm/fingers touch the foregrip.

Finger gripping is a separate stage using additive pose or Control Rig.

# Complete System Audit — Errors Found & Fixed

## CRITICAL ERRORS (Would crash or silently fail)

### 1. ❌ STTask_TakeCover: UAITask_MoveTo::AIMoveTo wrong signature
**File:** `STTask_TakeCover.cpp`
**Error:** `UAITask_MoveTo::AIMoveTo(AI, Cover.Location, AActor::StaticClass(), 80.f, ...)`
The second parameter should be `FVector` for location-based moves OR `AActor*` for actor-based.
Passing `AActor::StaticClass()` (a UClass*) in the target actor slot is wrong.
**Fix:** Use the location-only overload:
```cpp
UAITask_MoveTo::AIMoveTo(AI, Cover.Location, nullptr, 80.f);
```

### 2. ❌ STTask_FollowLeader: Same UAITask_MoveTo wrong signature
**File:** `STTask_FollowLeader.cpp`
**Error:** Same issue — `AActor::StaticClass()` passed as target actor.
**Fix:** Use `nullptr` for location-based moves.

### 3. ❌ StateTree Tick signature mismatch
**File:** All StateTree tasks
**Error:** Task `Tick` declared as `Tick(FStateTreeExecutionContext&, const float)` 
but the base class may use `Tick(FStateTreeExecutionContext&, float)` (no const).
Different UE versions have different signatures.
**Fix:** Match exact base class signature. In 5.7:
`virtual EStateTreeRunStatus Tick(FStateTreeExecutionContext& Context, const float DeltaTime) const override;`
This IS correct for 5.7 based on the API docs. ✅ No change needed.

### 4. ❌ Context.GetOwner() returns different types in different UE versions
**File:** All StateTree tasks and evaluators
**Error:** In some 5.7 builds, `GetOwner()` returns `const UObject*`, not `AActor*`.
**Fix:** Use the pattern from the reference doc:
```cpp
AActor* OwnerActor = Cast<AActor>(Context.GetOwner());
```
The `const_cast` in my code (`const_cast<AActor*>(Cast<AActor>(Context.GetOwner()))`)
is needed because we get a const pointer but need to call non-const methods.
**Status:** Already using `Cast<AActor>(Context.GetOwner())` which handles the cast.
But the `const_cast` is ugly. Better pattern: store controller ref in instance data
on EnterState and reuse on Tick. ✅ Already doing this for cover (via controller).

### 5. ❌ CoverScanner: FTraceDelegate vs FAsyncTraceDelegate naming
**File:** `CoverScanner.cpp`
**Error:** Used `FTraceDelegate` but the correct UE 5.7 type might be `FTraceDelegate`
or `FAsyncTraceDelegate` depending on the trace function used.
`AsyncLineTraceByChannel` uses the 6th parameter as `FCollisionResponseParams`
and the 7th as `FTraceDelegate*`.
**Fix:** Verify the parameter order matches the actual UE 5.7 API.

### 6. ❌ CoverScanner: BindRaw with member function and extra params
**File:** `CoverScanner.cpp`
**Error:** `Delegate.BindRaw(this, &FCoverScanner::OnAsyncTraceComplete, ChunkKey, SampleIndex, d, h)`
`FTraceDelegate` is a `TDelegate<void(const FTraceHandle&, FTraceDatum&)>`.
Extra payload parameters must be bound differently — UE delegates don't support
arbitrary extra params in BindRaw like std::bind.
**Fix:** Use a wrapper approach — store pending trace info in a TMap keyed by
FTraceHandle, look it up in the callback.

### 7. ❌ WaveSpawner: HandleSpawnedDied expects AActor* but OnDeath broadcasts AActor*
**File:** `WaveSpawner.cpp`
**Error:** `HC->OnDeath.AddDynamic(this, &AWaveSpawner::HandleSpawnedDied)` 
but `FOnHealthDied` broadcasts `AActor*` while `HandleSpawnedDied` takes `AActor*`.
This IS correct — both are `AActor*`. ✅ No issue.

### 8. ❌ EnemyPool: WakeEnemy calls SpawnDefaultController but controller class unknown
**File:** `EnemyPool.cpp`  
**Error:** `Enemy->SpawnDefaultController()` works but the spawned controller won't have
the StateTree asset assigned. The AI controller needs its StateTree reference.
**Fix:** Instead of SpawnDefaultController, re-possess with the original controller
(store it before sleeping) or ensure the pawn's AIControllerClass is set correctly.

### 9. ❌ QuestObjective: UObject can't use GetWorld() directly
**File:** `QuestObjective.h/.cpp`
**Error:** `UQuestObjective` is a `UObject`, not an `AActor`. Plain UObjects don't have
`GetWorld()` unless they have an outer that provides world context.
**Fix:** Pass `UWorld*` as parameter to `OnStart` and `Tick` (already doing this ✅).

### 10. ❌ MissionMinimapWidget: NativePaint signature wrong
**File:** `MissionMinimapWidget.h`
**Error:** `NativePaint` uses `FSlateClippedElement` which doesn't exist.
The correct UE 5.7 signature is:
```cpp
virtual int32 NativePaint(const FPaintArgs& Args, const FGeometry& AllottedGeometry,
    const FSlateRect& MyCullingRect, FSlateWindowElementList& OutDrawElements,
    int32 LayerId, const FWidgetStyle& InWidgetStyle, bool bParentEnabled) const override;
```
**Fix:** Change `FSlateClippedElement` to `FSlateRect`.

### 11. ❌ MissionMinimapWidget: FSlateDrawElement::MakeBox uses wrong style reference
**File:** `MissionMinimapWidget.cpp`
**Error:** `FAppStyle::Get().GetWidgetStyle<FButtonStyle>("Button").Normal` — this
might not exist or might be named differently. For simple box drawing, use
`FSlateColorBrush` or the `FCoreStyle` defaults.
**Fix:** Use a simpler brush approach.

### 12. ❌ CompassWidget: Lambda inside NativeTick captures nothing for English cardinals
**File:** `CompassWidget.cpp`  
**Error:** The lambda for English cardinals captures `PlayerYaw` but it's a

## 0. High-Level Integration Goal

The correct direction for this project is:

```text
SquadAI owns:
- mission logic
- leader/follower behavior
- enemy logic
- threat memory
- cover selection
- perception/team logic
- quest/zone logic
- high-level commands: move, aim, attack, stop

TPWCS pack owns:
- weapon inventory
- weapon giving/equipping
- firing execution
- trace start/end from weapon/barrel
- ammo/reload
- weapon spread/damage setup
- weapon mesh visibility/equip notify flow
- aiming state
- weapon animation state
- fire/reload/equip montages
- hand IK/weapon pose where applicable
- pack health/ragdoll/melee behavior where used
```

The most important rule:

```text
Do not recreate TPWCS weapon/animation internals in SquadAI if the pack already does it.
```

Instead, use working pack Blueprints as templates and attach SquadAI logic around them.

## 2. Recommended Generated Blueprint Architecture for SquadAI + TPWCS

The safest current architecture is to duplicate TPWCS pack templates while preserving the pack parent chain.

Recommended enemy chain:

```text
BP_Enemy_PackClean_Auto
    parent: BP_Enemy_PackClean_Auto_PackBase
        parent: AEnemySoldier
            parent: ASoldierCharacter
                parent: ACharacter
```

But `BP_Enemy_PackClean_Auto` should be created from:

```text
BP_AICharacter
```

And `BP_Enemy_PackClean_Auto_PackBase` should be created from the pack parent:

```text
BP_Character
```

This keeps pack parent Blueprint logic while still making the final generated BP `IsA(AEnemySoldier)`.

Same idea:

```text
BP_Ally_PackClean_Auto
    parent: BP_Ally_PackClean_Auto_PackBase
        parent: AAllySoldier

BP_Leader_PackClean_Auto
    parent: BP_Leader_PackClean_Auto_PackBase
        parent: ALeaderCharacter
```

Do not directly reparent `BP_AICharacter` to `AEnemySoldier`, because that risks losing `BP_Character` pack parent graph/defaults.

# 10. Additional Findings From Older Project Notes

## 10.1 Original project state before TPWCS integration

Older handoff notes describe the project as a custom Unreal C++ system containing:

```text
- custom AI controller logic
- perception memory
- direct combat fallback
- cover scanning and scoring
- quest/zone mission logic
- leader pathing and ally follow
- enemy attack coordination
- custom health/damage/death handling
- previous custom weapon hitscan handling
- previous custom animation montage handling
- Character Creator / Marvelous Designer character pipeline
```

The AI/cover/combat side was already functional enough that enemies and allies could:

```text
- move to mission/quest zones
- detect hostiles
- take cover
- fire
- damage and kill enemies
- stop dead enemies from continuing to fight
```

Therefore TPWCS is not replacing the whole game architecture. It is mainly being used to replace the fragile weapon/animation/presentation layer.

## 10.2 Why Lyra was removed

`LYRA_REMOVAL_STAGE1_NOTES.md` explains that the project moved away from Lyra because it added too much dependency/complexity for the current goal. Important changes included:

```text
ASoldierCharacter now inherits from ACharacter instead of ALyraCharacter.
LyraPawnExtensionComponent setup was removed.
Lyra HeroData loading was removed.
Lyra health/death event hookup was removed.
Lyra-specific PossessedBy logic was removed.
Lyra-related controller/spawner casts/comments were removed.
LyraGame / ModularGameplayActors dependencies were removed.
```

Implication for TPWCS:

```text
Do not introduce another giant framework conflict by mixing TPWCS player systems with SquadAI low-level weapon/aim systems.
Keep TPWCS as the combat presentation/execution layer, and keep SquadAI as high-level game logic.
```

## 10.3 Safe pack integration rules discovered earlier

`PACK_SAFE_INTEGRATION_NOTES.txt` established several rules that are still valid:

```text
Goal:
- Keep SquadAI logic.
- Do not let SquadAI visual/animation systems override or damage the pack system.
- Make setup as drag-and-drop as possible.
```

## 10.4 Player versus AI boundary

Older notes warn that the pack player character should not share SquadAI aim/fire logic.

Correct boundary:

```text
Player:
- use pack player BP
- use pack input
- use pack weapon system
- use pack aiming/UI/camera
- use pack animation
- do not add SquadAI AimComponent/WeaponComponent behavior to player

AI / Leader / Allies:
- use SquadAI mission/cover/perception/team logic
- use TPWCS character/weapon/presentation pieces
- use bridge only for high-level aim/fire/equip calls
```

Reason:

```text
AimComponent directly rotating characters/controllers can conflict with pack player camera/aim/UI behavior.
```

Older audit notes later reduced this risk by making controller rotation side effects opt-in, but the architecture boundary is still important.

## 10.5 Controller-level combat API was created for pack integration

The scratch factory attempted to create a new BP from `AEnemySoldier` / `AAllySoldier` / `ALeaderCharacter` and add TPWCS components manually.

```text

- Blueprint parent logic
- Blueprint interface implementation
- data table row IDs
- animation BP expected state
- equip notify timing
- weapon mesh and socket defaults
- event dispatchers
- controller/detection assumptions
```

Therefore the current design prefers:

```text

Only add SquadAI helpers after preserving pack setup.
```


## 10.9 Quest objective safety discovered earlier

Older notes state that quest objective logic was changed so it does not push mission goals to every AI controller globally.

Current rule:

```text

```

This matters because enemy AI should not be redirected by player/leader quest flow unless explicitly intended.

## 10.10 Damage bridge principle

The pack may apply damage through its own hit logic. SquadAI uses its own `HealthComponent` and death/registry logic.

Bridge principle:

```text
Pack hit event
→ ForwardPackDamageToSquadAI(HitActor, DamageAmount, DamageCauser, InstigatorController, DamageTypeClass)
→ ASoldierCharacter::NotifyPackDamage
→ HealthComponent::ApplyExternalDamage
→ SquadAI death/registry/perception side effects
```

Do not rely on both systems independently killing the same actor without a bridge, because that can cause duplicated death/ragdoll/state problems.

## 10.11 Remaining AI/cover concerns independent of TPWCS

Older `CURRENT_STATE_AND_TODO.md` notes that AI/cover is mostly functional, but still needs polish:

```text
- cover selection can still be imperfect
- engage state may shoot into landscape/sandbags if selected cover blocks line of sight
- blocked-shot detection should request better cover/peek position
- leader behavior should be regression-tested after animation/presentation changes
```

TPWCS may improve weapon presentation but does not automatically solve tactical cover scoring.

## 10.12 Old logical muzzle versus TPWCS barrel trace

Older SquadAI weapon code introduced a logical AI muzzle because animation/hand pose was not stable yet:

```text

LogicalMuzzleForwardOffset
```

This was useful before TPWCS. But for TPWCS integration it becomes wrong if the goal is to use the pack’s weapon/barrel trace.

Current pack-mode rule:

```text
SquadAI old logical muzzle must be disabled/bypassed.

```

# 13. Failure Mode Catalog

## 13.1 Weapon appears in hand but hand placement is wrong

Likely causes:

```text
- weapon was given but not fully equipped through pack equip flow
- equip montage notify ShowWeaponMesh did not run
- wrong weapon row / wrong slot selected
- active Anims Set ID not matching DT_WeaponAnims row
- AI is not in aiming state, so aiming/IK pose is inactive
- AI left-hand IK disabled by default
- generated BP lost pack parent logic/defaults
- duplicated wrong template, e.g. Blank/Test instead of normal BP_AICharacter
```

Best diagnosis:

```text
Compare working BP_AICharacter runtime BPC_WeaponCharacterAI values with generated BP runtime values.
```

## 13.2 Enemy/ally fires from head/top instead of weapon

Cause:

```text
Old SquadAI WeaponComponent StartBurst/FireAtTarget path is still active.
```

Fix:

```text
Pack mode must make old WeaponComponent return false / do nothing.
Adapter must call TPWCS fire entry.
```

## 13.3 AI detects/uses cover but does not fire

Possible causes:

```text
- bridge did not find exact TPWCS fire function
- pack requires BPC_AIDetection_Master / AI detection state
- pack weapon has no active weapon/equipped slot
- pack weapon says not usable / no ammo / not aiming
- SquadAI stops presentation too early during cover state transitions
```

Debug:

```text
Look for [SquadAIAdapterPackBridge] warnings.
Inspect BPC_WeaponCharacterAI Active Weapon / Is Shooting? / Ammo.
```

## 13.4 Generated BP compile errors after duplication

Possible causes:

```text
- reparenting pack BP_Character to AEnemySoldier broke inherited variable/function assumptions
- duplicate PackBase missing expected parent class functions
- Blueprint graph references original parent-only variables incompatible with new parent
```

Fallback options:

```text
1. Keep generated BP parent as original pack BP and adapt SquadAI through interfaces/components.
2. Create native C++ subclass that mimics the pack parent requirements.
3. Use scratch factory only for logic tests, not final presentation.
```

## 13.5 Player UI/aim breaks when AI aims

Possible cause from older notes:

```text
Shared or generic AimComponent/controller rotation logic affecting pack player state.
```

Rule:

```text
Do not put SquadAI combat presentation components on the pack player.
Keep player entirely pack-owned.
```
